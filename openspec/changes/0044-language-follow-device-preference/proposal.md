# Proposal: 语言跟随设备当前偏好

## 背景 / 动机

- 当前项目把语言选择建立在地区码优先规则上，导致设备语言与地区不一致时，App 仍显示地区对应语言。
- 用户实际期望是：手机当前语言切到日语时，App 立即显示日语，而不应继续受地区码影响。
- 需要将语言选择逻辑调整为优先跟随设备当前语言偏好，同时保留地区码用于埋点记录。

## 目标（Goals）

- G1: App 启动时优先根据设备当前语言偏好选择本地化资源。
- G2: 当设备语言为日语、韩语、英语、简体中文时，App 应直接切换到对应语言，与地区码无关。
- G3: 地区码继续仅用于 CloudKit 启动埋点，不再决定 UI 语言。

## 非目标（Non-goals）

- NG1: 不改 CloudKit 启动埋点字段结构。
- NG2: 不扩展四种语言之外的新语言资源。
- NG3: 不新增语言切换 UI。

## 范围（Scope）

- **会改的部分**：
  - 文件/模块：`AppLanguageManager.swift`、本地化主规格、OpenSpec 变更文档。
  - 行为变化：语言选择由地区码优先改为设备语言优先。
- **不会改的部分**：
  - `AppLaunchEvent` 中的 `regionCode` 字段含义。
  - 已有本地化字符串内容。

## 需求概要

- App 必须优先读取设备当前语言偏好，而不是地区码，决定本次运行的本地化语言。
- 支持语言范围仍为英语、日语、韩语、简体中文。
- 当设备语言不在支持范围内时，才回退到应用支持语言中的最佳匹配，最终兜底英文。
- 地区码仅用于日志与埋点，不得再覆盖用户当前语言。

## 验收标准（Acceptance Criteria）

- AC1: 设备语言切换为日语时，即使地区仍为中国，App 核心文案也显示为日语。
- AC2: 设备语言切换为韩语、英语、简体中文时，App 同样切到对应语言。
- AC3: 不支持的设备语言下，App 会安全回退到支持语言中的最佳匹配或英文。
- AC4: CloudKit 启动埋点仍继续上报地区码，但不影响 UI 语言选择。

## 风险与回滚

- **风险**：若设备语言解析逻辑选错优先级，可能导致回退语言异常。
- **回滚方案**：恢复 `AppLanguageManager` 的地区码优先映射规则，并同步回退规格描述。
