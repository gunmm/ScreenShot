# Proposal: 涂抹后进入拼接调整前警告

## 背景 / 动机

- 当前主界面允许用户在完成涂抹/打码后继续进入“拼接调整”。
- 但拼接调整会基于原始分片和裁剪区间重新生成长图，不会保留已经合成到结果图上的涂抹内容。
- 这会让用户误以为之前的涂抹仍然有效，直到重新拼接后才发现效果丢失。

## 目标（Goals）

- G1: 在当前结果图已经做过涂抹时，用户点击“拼接调整”前收到明确提示。
- G2: 提示文案明确说明继续拼接调整会丢弃已涂抹效果，由用户自行确认是否继续。

## 非目标（Non-goals）

- NG1: 不实现涂抹层与拼接裁剪区间的联动重算。
- NG2: 不改变涂抹导出、保存和水印逻辑。

## 范围（Scope）

- 会改的部分：
  - 文件/模块：主界面编辑入口与状态管理。
  - 行为变化：当当前结果图包含涂抹修改时，进入拼接调整前弹出确认框。
- 不会改的部分：
  - 拼接算法本身。
  - PencilKit 涂抹编辑器内部交互。

## 需求概要

- 主界面需要识别当前预览图是否基于拼接结果做过涂抹。
- 若用户在该状态下点击“拼接调整”，必须先弹出确认框。
- 确认框必须明确提示继续操作会丢弃已涂抹效果。
- 用户取消后，保留当前带涂抹的预览图和状态不变。
- 用户确认后，按现有流程进入拼接调整。

## 验收标准（Acceptance Criteria）

- AC1: 未做过涂抹时，点击“拼接调整”直接进入编辑页，无新增提示。
- AC2: 做过涂抹后，首次点击“拼接调整”弹出提示框，文案明确说明会丢弃已涂抹效果。
- AC3: 提示框取消后，当前结果图保持不变，仍可继续保存或再次涂抹。
- AC4: 提示框确认后，进入现有拼接调整页，后续重新拼接结果覆盖当前结果图。

## 风险与回滚

- 风险：若“已做过涂抹”的标记维护不准确，可能导致提示缺失或误提示。
- 回滚方案：移除主界面的标记字段和确认弹窗逻辑，恢复当前直接进入拼接调整的行为。
