# Design: 涂抹后进入拼接调整前警告

## 现状（As-is）

- `ViewController` 用 `state.generated(images, ranges)` 记录当前拼接来源分片和裁剪区间。
- `markupResult()` 会把涂抹后的合成图写回 `rawStitchedImage` 和主界面预览。
- `editResult()` 进入拼接调整时只依赖 `state.generated` 中的 `images` 与 `ranges`，重新拼接后会覆盖 `rawStitchedImage`。
- 因此，只要用户在涂抹后再次进入拼接调整，之前的涂抹效果必然丢失。

## 方案（To-be）

- 总体思路：
  - 在主界面维护一个布尔标记，表示“当前预览图是否包含用户涂抹修改”。
  - 只有在该标记为真时，点击“拼接调整”先展示确认框。
  - 用户确认后继续执行原有编辑入口；取消则不进入编辑页。
- 模块与职责：
  - `ViewController`：维护标记、判断是否弹窗、承接确认后的跳转。
- 数据流：
  1. 自动拼接或手动拼接成功后，清空“已涂抹”标记。
  2. 用户完成涂抹并确认后，将标记置为真。
  3. 用户点击“拼接调整”时：
     - 标记为假：直接进入编辑页。
     - 标记为真：先弹出警告，确认后进入编辑页。
  4. 用户完成拼接调整并重新生成结果图后，标记重置为假。

## 接口/数据结构变更

- `ViewController` 新增布尔状态字段，用于记录当前结果图是否已应用涂抹。
- `ViewController.editResult()` 拆分出实际进入编辑页的私有方法，供直接路径和确认后的路径复用。

## 关键算法/阈值

- 无新增算法。
- 警告只在“当前结果图已涂抹”这一条件下触发，不增加次数限制或额外持久化。

## 兼容性与迁移

- 不涉及数据持久化变更。
- 旧版本录制分片、拼接结果和保存流程不受影响。

## 可观测性

- 本次不新增日志；问题可通过界面路径直接复现。

## 回滚策略

- 删除标记字段和确认弹窗逻辑，恢复当前编辑入口实现即可。
