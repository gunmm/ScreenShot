# Proposal: 启动埋点与国家语言适配

## 背景 / 动机

- 当前 CloudKit 仅覆盖用户反馈和保存成功后的自动日志上传，缺少应用启动维度的基础运行样本。
- 现有多语言资源只覆盖中文和英文，且部分用户可见文案仍直接写死在代码中，无法满足美国、日本、韩国、中国四个国家的完整语言适配。
- 需要在不引入后端和账号体系的前提下，为启动时的基础运营数据和国家语言适配提供最小可用实现。

## 目标（Goals）

- G1: App 每次启动时向 CloudKit Public Database 异步写入一条启动记录。
- G2: 启动记录必须包含 userId、appVersion、是否已付费、地区码、时间。
- G3: App 必须根据当前设备地区码，在美国、日本、韩国、中国四个国家间分别切换为英文、日文、韩文、简体中文。
- G4: 用户可见核心文案在上述四个国家环境下必须有对应本地化结果，不能继续回退为中文硬编码。

## 非目标（Non-goals）

- NG1: 不实现埋点批量补传、失败重试队列或统计聚合后台。
- NG2: 不扩展到四国外的全部语种适配。
- NG3: 不重构所有历史页面的 UI 布局，仅修复与本次四国语言适配直接相关的文案与语言选择逻辑。

## 范围（Scope）

- **会改的部分**：
  - 文件/模块：AppDelegate 启动流程、CloudKitManager、付费状态读取、多语言资源目录、部分写死文案页面、主规格文档。
  - 行为变化：启动后异步上传一条启动记录；应用根据地区码自动选择中英日韩文案。
- **不会改的部分**：
  - 保存成功后的自动日志上传频控策略。
  - 支付商品和 CloudKit 容器配置本身。
  - 四国外地区的单独语言策略。

## 需求概要

- App 冷启动或热启动进入 didFinishLaunching 阶段时，必须异步触发一次 CloudKit 启动记录上传。
- 启动记录必须使用独立的 CloudKit record type，且至少包含 userId、appVersion、isPaid、regionCode、launchedAt。
- 启动记录上传失败不得阻塞应用启动，也不得向用户弹错。
- App 启动时必须根据地区码映射语言：US -> en，JP -> ja，KR -> ko，CN -> zh-Hans。
- 若地区码不在映射范围内，系统应回退到现有支持语言中的最佳匹配，不得崩溃。
- 原本写死在代码中的核心用户可见文案必须切到本地化资源，至少覆盖设置、打赏、购买失败与恢复相关文案。

## 验收标准（Acceptance Criteria）

- AC1: 每次应用启动时，CloudKit Public Database 中会新增一条启动记录，字段包含 userId、appVersion、isPaid、regionCode、launchedAt。
- AC2: 启动记录上传在后台异步执行，失败仅记本地日志，不影响应用进入主界面。
- AC3: 设备地区为美国、日本、韩国、中国时，应用核心文案分别展示英文、日文、韩文、简体中文。
- AC4: 日本和韩国环境下，打赏、购买失败、恢复购买、设置页等核心入口不再显示中文硬编码。
- AC5: 非四国地区环境下，应用仍可正常启动，并回退到现有支持语言中的最佳匹配。

## 风险与回滚

- **风险**：若地区语言切换过早或过晚，可能导致首屏文案未生效；若新增 CloudKit 记录字段类型不一致，可能导致写入失败。
- **回滚方案**：移除 AppDelegate 中的启动上传入口与语言注入逻辑，删除新增启动记录写入代码和新增语言资源，恢复到现有系统语言回退行为。
